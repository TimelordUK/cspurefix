KEY ARCHITECTURAL PATTERNS:
============================

PARSING PATTERN:
  QuickFixXmlFileParser → Graph construction → Work queue → ContainedFieldSet → Index visitor

DEFINITION vs RUNTIME vs GENERATED:
  Definition (metadata) → Contained (with context) → Generated C# (user code)

VISITOR PATTERN:
  ISetDispatchReceiver → OnSimple/OnComponent/OnGroup (used by IndexVisitor, ContainedFieldCollector)

GENERATION PATTERN:
  FixDefinitions → MessageGenerator.Process() → CodeGenerator → Write files

CODE ORGANIZATION BY RESPONSIBILITY:
  Parser/ - XML parsing and graph building
  Definition/ - Metadata structures
  Contained/ - Runtime structures with indices
  Compiler/ - C# code generation

CRITICAL FILES (for understanding flow):
  1. /Parser/QuickFix/QuickFixXmlFileParser.cs - Entry point for parsing
  2. /Definition/FixDefinitions.cs - Container and lookup
  3. /Contained/ContainedFieldSet.cs - Runtime structure
  4. /Compiler/MessageGenerator.cs - Code generation
  5. /PureFix.App/Program.cs - CLI entry point

PROBLEMATIC FILE:
  /Parser/FieldEnum.cs - Line 18 has no-op replace() call

DISABLED COMPONENTS:
  /Compiler/MsgCompiler.cs - Alternative generator (not compiled)

LARGE FILES (>100 lines):
